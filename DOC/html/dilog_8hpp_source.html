34/* Real part of DiLogarithm(x), for real argument.
35 * In Lewin's notation, this is Li_2(x).
36 *
37 * Li_2(x) = - Re[ Integrate[ Log[1-s] / s, {s, 0, x}] ]
38 *
39 * The function in the complex plane has a branch point
40 * at z = 1; we place the cut in the conventional way,
41 * on [1, +infty). This means that the value for real x > 1
42 * is a matter of definition; however, this choice does not
43 * affect the real part and so is not relevant to the
44 * interpretation of this implemented function.
45 */
46double dilog(double x);

49/* DiLogarithm(z), for complex argument z = x + i y.
50 * Computes the principal branch.
51 *
52 * Recall that the branch cut is on the real axis with x > 1.
53 * The imaginary part of the computed value on the cut is given
54 * by -Pi*log(x), which is the limiting value taken approaching
55 * from y < 0. This is a conventional choice, though there is no
56 * true standardized choice.
57 *
58 * Note that there is no canonical way to lift the defining
59 * contour to the full Riemann surface because of the appearance
60 * of a "hidden branch point" at z = 0 on non-principal sheets.
61 * Experts will know the simple algebraic prescription for
62 * obtaining the sheet they want; non-experts will not want
63 * to know anything about it. This is why GSL chooses to compute
64 * only on the principal branch.
65 */
66double complex_dilog_xy(double x, double y, _re, _im);

70/* DiLogarithm(z), for complex argument z = r Exp[i theta].
71 * Computes the principal branch, thereby assuming an
72 * implicit reduction of theta to the range (-2 pi, 2 pi).
73 *
74 * If theta is identically zero, the imaginary part is computed
75 * as if approaching from y > 0. For other values of theta no
76 * special consideration is given, since it is assumed that
77 * no other machine representations of multiples of pi will
78 * produce y = 0 precisely. This assumption depends on some
79 * subtle properties of the machine arithmetic, such as
80 * correct rounding and monotonicity of the underlying
81 * implementation of sin() and cos().
82 *
83 * This function is ok, but the interface is confusing since
84 * it makes it appear that the branch structure is resolved.
85 * Furthermore the handling of values close to the branch
86 * cut is subtle. Perhap this interface should be deprecated.
87 */
88double complex_dilog(
89 double r,
90 double theta,
91 _re,
92 _im
93 );
